524  class xorshift1024_star
525  {
526  /* No doubt an SIMD version of this generator would be faster for massively parallel sampling.
527  * This would produce 16 new random words in one large batch, bu would require doubling the state
528  * s0 = current state
529  * s1 = last state
530  * We seed the generator as normal
531  * s0 = seed
532  * s1 = s0 (rotated left one index) ... for later use
533  * To warm up, we run the generator 16 times (using state = s0 and the original algorithm).
534  * This makes sure s0 has incremented in every index. Now every time we run the generator,
535  * we replace s0 -> s0[i] and s1 -> s1[i], then loop i = [0,16), which should be auto-vectorized.
536  * Afterwards, s0 is unchanged, s1 stores the new state, which we copy to s2 for return.
537  * Now s1 = s0 (rotated left one index) and s0 = s2 (less copying by swapping pointers, but maybe not faster).
538  * I'm pretty sure this is what Lemire did in (https://github.com/lemire/SIMDxorshift),
539  * and he reported a doubling of the sampling speed. I would expect more than doubling,
540  * given the 4 times larger state and increased vectorized advantage.
541  * However, speeding up the PRNG will just further highlight that
542  * the real bottleneck is transforming the uniform integers into useful numbers,
543  * because it takes a lot of cycles and generally involves
544  * branching logic in the various math functions (i.e. I'm not aware of
545  * good, universal SIMD exp, log, pow with 1 ULP or error).
546  */
547  public:

550  size_t static constexpr state_size = 16;
