Detailed Description

An implementation of the xorshift1024* 64-bit pseudo-random number generator (PRNG).

Author
Sebastiano Vigna (vigna.nosp@m.@acm.nosp@m..org)
Keith Pedersen (Keith.nosp@m..Dav.nosp@m.id.Pe.nosp@m.ders.nosp@m.en@gm.nosp@m.ail..nosp@m.com)

KP's implementation of SV's xorshift1024* (period \( p = 2^{1024} - 1\)), with small changes to the code syntax and a new API that mimics std::mt19937.

SV claims that the generator has good quality and speed, passing TestU01 with less errors than MT19937. KP independently tested this implementation using dieharder 3.31.1

./xorshift2014star_tester.x | dieharder -g 200 -a -k 2

and it passed with flying colors (no failures).

Attention
Per SV: "... The three lowest bits of this generator are LSFRs, and thus they are slightly less random than the other bits. We suggest to use a sign test to extract a random Boolean value."

The last 3 bits also passed dieharder (concatenating the last-3 bits from three sequential calls)

./xorshift2014star_last3bits_tester.x | dieharder -g 200 -a

so perhaps they're not as bad as SV implies. But for now, we conservatively do not use the last 3 bits inside pqRand::engine.

Member Function Documentation

void pqRand::xorshift1024_star::Jump ( )

Quickly jump the state of the generator forward by \( 2^{512} \) calls.

This is useful when many instances of xorshift1024_star are running in parallel threads, since feeding each its own random seed might accidentally cause the sequences from one pair of generators to overlap, thus become correlated (combinatorially, this is too probable and too calamitous to ignore). Alternatively, storing the seeds from every thread in every simulation — for auditing purposes — would require a lot of storage.

The Jump() method only requires storing one seed per simulation:

size_t const numThreads = 16; // A simulation with 16 threads
std::vector<pqRand::engine> engineVec; // A vector storing each threads generator
engineVec.emplace_back(); // Auto-seed the first generator ...
engineVec.back().WriteState("mySimulationID.seed"); // then store its seed.
// Now create a bunch of orthogonal generators
for(size_t threadIndex = 1; threadIndex < numThreads; ++threadIndex)
{
engineVec.emplace_back(engineVec.back()); // Each engine starts as a copy of the last engine ...
engineVec.Jump(); // but is jumped one additional time
}
Note
Obviously, the state of the generator is jumped without actually calling it \( 2^{512} \) times (since, at a billion calls per second, that would take \( 4\times10^{137} \) years ... a bit too long).
Attention
This method seems like much deeper magic than the generator itself, so KP cannot guarantee that it actually works as advertised by SV. Howerver, after the Jan 2016 patch , it is definitely commutative [(call, Jump, call) == (Jump, call, call)], which certainly implies that it works.

Output the complete state-string of the generator to a stream.

The state-string is an ASCII representation of the PRNG's internal state. In pqRand, the minimal state-string mimics GNU's implementation of std::mt19937: a single line containing space-separated words of state, with the number of words equalling state_size, and terminating with the state_size itself.

s_0 s_1 s_2 ... s_(state_size - 1) state_size // Generic minimal state-string
s_0 s_1 s_2 ... s_15 16 // Minimal state-string for xorshift1024*

An example PRNG, using word_size = 64 and state_size = 3, could use the following minimal state-string:

8602057372317241997 13337802638347508439 10520579822156476364 3

A PRNG may contain additional components of its internal state which are not accounted for in state_size. For xorshift1024_star, the exact state includes an index iterator p (where p < state_size). Thus, its complete state-string is:

s_0 s_1 s_2 ... s_15 16 p // Complete state-string for xorshift1024*

Because p < state_size, it should not be determined from a random source of entropy, but from the number of times the generator is called (modulus 16). This is why it is not counted towards state_size.

Seed the state of the generator from a stream, assuming the state-string format of operator<<().

If p is missing from the state-string, a default value of p = 0 is used (see uPRNG_64_Seeder::Seed()).

Warning
Throws an exception if the stream runs out before the state is full, or if the stream does not follow the expected format.
Per SV: "The state must be seeded so that it is not everywhere zero."
Nonetheless, the only checks made by operator<<() on a supplied state-string is that it can be parsed in the expected format. When generating a user-defined seed (instead of using an auto-seed), take care not to supply a bad seed.
